# Changelog

## Unreleased

### Added
- **K8s Manage (Phase 8-11)** — security hardening, broader resource/CRD coverage, bulk actions + cluster overview, and a bridge from the older env-diff tool to full-manifest diffing:
  - **Phase 8 — Security/RBAC**: Secret YAML is redacted by default (`***REDACTED***` per key) with an explicit "Reveal secret values" toggle in the drawer; new RBAC kinds (ServiceAccounts, Roles, RoleBindings, ClusterRoles, ClusterRoleBindings); a new "Access" drawer tab runs a `SelfSubjectAccessReview` per verb (get/list/watch/create/update/patch/delete) so you can see exactly what the current identity can do on any resource.
  - **Phase 9 — Breadth + CRDs**: NetworkPolicies, StorageClasses, ResourceQuotas, LimitRanges added via the existing generic per-kind pattern; a new "Custom Resources" sidebar section discovers installed CRDs (`ApiextensionsV1Api`) and browses their instances generically via `CustomObjectsApi`, keyed by group/version/plural rather than a hardcoded kind — a parallel code path from the built-in 21 kinds, with YAML/Events/Delete support.
  - **Phase 10 — Bulk actions, search, overview**: row checkboxes + a bulk action bar (Restart/Delete across a selection, bounded concurrency, partial-failure summary banner); a "Search all kinds" fan-out search across every resource kind at once; a new Overview landing page (cluster-wide health digest: pods not Ready, unhealthy Deployments, NotReady Nodes, recent Warning events) that's the default view on entering Manage.
  - **Phase 11 — Env-diff bridge**: the older K8s ENV Diff screen gains a "Full Manifest" tab alongside "Env Vars", diffing both sides' full Deployment YAML (reusing `get-resource-yaml`, no new IPC) with an optional "Hide status differences" toggle; vendors `diff` (jsdiff) the same way `@xterm` is vendored (no bundler).
  - `main.js`: new `check-access`, `search-resources`, `get-manage-overview`, `list-crds`, `list-custom-resource`, `get-custom-resource-yaml`, `get-custom-resource-events`, `custom-resource-action` IPC handlers; `list-resource`'s per-kind dispatch extracted into a shared `listKindItems` helper reused by search.
  - `preload.js`: exposes all of the above.
  - `renderer/index.html` / `renderer/app.js` / `renderer/styles.css`: new sidebar sections, drawer tab, bulk bar, search-results panel, overview tiles, CRD browser, and the diff-view tab strip + manifest pane.
- **K8s Manage (Phase 4)** — read-only YAML manifest view + resource-scoped Events, available for every resource kind:
  - Drawer gains "YAML" and "Events" tabs (between Detail and Logs). YAML fetches the live object via the matching `read*`/`readNamespaced*` call, strips `metadata.managedFields`/`resourceVersion`, and renders it with `k8s.dumpYaml` in a monospace pane with a Copy button. Events filters `listNamespacedEvent` by `involvedObject.name`/`involvedObject.kind`, sorted newest-first; for Nodes (cluster-scoped, but their Events live in a namespace that may differ from the UI's selected namespace) it searches across all namespaces via `listEventForAllNamespaces` instead.
  - `main.js`: `get-resource-yaml`/`get-resource-events` IPC handlers, following the existing `{ok,...}` convention so an RBAC-denied read renders inline instead of crashing.
  - `preload.js`: exposes `getResourceYaml`/`getResourceEvents`.
  - `renderer/app.js`: both tabs fetch on-demand (one-shot, not polled) when clicked, and drop stale responses if the selected row/kind changed while the request was in flight — same guard pattern as `refreshManageMetrics`.
- **K8s Manage (Phase 5)** — safe, confirm-gated resource actions; no arbitrary YAML edit/apply:
  - Drawer header gains per-kind action buttons: rollout **Restart** (Deployments/StatefulSets/DaemonSets), **Scale** replicas (Deployments/StatefulSets), **Cordon/Uncordon** (Nodes, label flips based on `spec.unschedulable`), and **Delete** for every kind (requires typing the resource name to enable the button).
  - `main.js`: one guarded `resource-action` IPC handler with an explicit allow-list (`restart|scale|delete|cordon|uncordon`); mutations use `patchNamespaced*`/`patchNamespaced*Scale`/`patchNode` with the `application/strategic-merge-patch+json` content-type header (a 415 gotcha if omitted), and rollout restart uses the same `kubectl.kubernetes.io/restartedAt` annotation trick `kubectl rollout restart` relies on. RBAC-denied actions return `{ok:false,error}` instead of throwing.
  - `preload.js`: exposes `resourceAction`.
  - `renderer/app.js`: a reusable `showManageConfirm()` modal (new `#manage-confirm-overlay`, styled after the existing `.auth-overlay`/`.auth-card`) supports a plain yes/no confirm, a type-the-name gate for deletes, or a numeric replica-count input for scale. On success the resource list refreshes immediately (and the drawer closes on delete); failures surface via the existing `alert()` convention used elsewhere in the app.
  - `renderer/styles.css`: `.manage-actions`, `.btn-danger`, `.manage-confirm-*`.
- **K8s Manage (Phase 6)** — local port-forwarding to a pod:
  - Pod drawer gains a "Port-forward" tab: enter a pod port (and optional local port, else OS-assigned), Start, and the forward appears in an active-forwards list (`localhost:PORT → pod:PORT`) with a Stop button. Unlike Logs/Exec/Metrics, forwards are independent background proxies — they are **not** tied to the drawer, the active tab, or even the pod that started them; they keep running while browsing other resources and are only torn down by an explicit Stop, or by leaving the Manage workspace (or closing the app).
  - `main.js`: `pf-start`/`pf-stop` IPC handlers wrap `k8s.PortForward`; `pf-start` opens a local `net.Server`, and each incoming local TCP connection gets its own `portForward()` WebSocket (the accepted socket doubles as both the input and output stream — the client-node port-forward API only supports one target port per call). Sessions tracked in a `pfSessions` Map, closed on explicit stop and on `mainWindow 'closed'` alongside the existing log/exec teardown.
  - `preload.js`: exposes `startPortForward`/`stopPortForward` plus a disposer-returning `onPortForwardError` (surfaced to the user via `alert()`, matching the app's existing error convention).
  - `renderer/app.js`: `state.manage.portForwards` (Map, keyed by session id) backs the active-forwards list; `showView`'s teardown choke point now also calls `stopAllManagePortForwards()` when leaving `manage`.
  - `renderer/styles.css`: `.manage-pf-*`.
- **K8s Manage (Phase 7)** — broader resource coverage and all-namespaces browsing:
  - Sidebar gains ReplicaSets, Ingresses, Jobs, CronJobs, PVCs, HPAs, PVs, and Namespaces (17 kinds total), each wired through the same generic `list-resource`/`get-resource-yaml`/`get-resource-events`/`resource-action` (delete) handlers as the original 9 — no new IPC surface, just more `case`s per kind using `BatchV1Api`/`NetworkingV1Api`/`AutoscalingV2Api` alongside the existing `CoreV1Api`/`AppsV1Api`. PVs and Namespaces join Nodes as cluster-scoped (namespace selector ignored).
  - Namespace selector gains an "(All namespaces)" option. `main.js` recognizes a sentinel namespace value and switches every namespaced kind's list call to its `*ForAllNamespaces()` variant (including pod metrics, via `Metrics.getPodMetrics()` with no namespace arg); `projectRow()` now always includes a `namespace` field so the renderer can show a Namespace column only while browsing all-namespaces.
  - Per-row operations (YAML, Events, Logs, Exec, actions, Port-forward) resolve the concrete namespace from the **selected row**, not the header selector, when in all-namespaces mode — a new `manageRowNamespace(row)` helper in `renderer/app.js` centralizes this so every call site stays correct regardless of which namespace the browsed resource actually lives in.
  - `manageStatusClass()` extended to recognize additional healthy/transitional states (`Active`, `Bound`, `Available`, `Terminating`, `Released`) introduced by the new kinds, so e.g. a Namespace in `Active` phase doesn't render as an error pill.
  - Fixed a metrics-series collision that all-namespaces mode would otherwise expose: sparklines were keyed by pod/node **name** alone, so two identically-named pods in different namespaces (a common pattern, e.g. `web-0` in two environments) would silently overwrite each other's CPU/Memory chart. Both `get-metrics` and the client-side `metricsSeries` map now key by `namespace/name` via a shared `metricsKey()` helper.
- **K8s Manage (Phase 1)** — new home-screen tool for browsing a single cluster, Lens-style:
  - Resource list for Pods, Deployments, StatefulSets, DaemonSets, Services, ConfigMaps, Secrets, Nodes, and Events, with a sidebar switcher, client-side name filter, and auto-refresh polling (5s; 10s for Nodes/Events).
  - Detail drawer with a live pod log viewer: container picker, follow-tail toggle (auto-disables on manual scroll-up), configurable tail-line count, and a bounded ring buffer to cap memory use.
  - Cluster entry point reuses the existing AKS cluster picker, generalized with a `maxSelect` parameter (2 for K8s ENV Diff, 1 for K8s Manage) instead of duplicating the picker UI; also supports "Use local kubeconfig".
  - `main.js`: one generic `list-resource` IPC handler projects each resource kind down to the fields the table needs and returns `{ok, rows}` / `{ok:false, error}` so an RBAC-denied kind renders an inline error instead of crashing the poller; `start-pod-logs`/`stop-pod-logs` stream logs via `k8s.Log`, coalescing chunks into a single flush every 150ms and truncating past a 256KB buffer, with sessions aborted on window close.
  - `preload.js`: exposes `listResource`, `startPodLogs`/`stopPodLogs`, and disposer-returning `onPodLogData`/`onPodLogEnd`/`onPodLogError` listeners.
  - Leaving the Manage workspace (or closing the app) is a single teardown choke point that stops the poll timer and any open log stream, so nothing keeps running in the background.
- **K8s Manage (Phase 2)** — pod exec/shell in the detail drawer, via vendored `xterm.js` (no bundler, loaded as plain `<script>`):
  - New "Exec" tab opens an interactive shell in the selected pod (tries `/bin/bash`, falls back to `/bin/sh`), with a container picker for multi-container pods.
  - `main.js`: `exec-start`/`exec-write`/`exec-resize`/`exec-stop` IPC channels drive `k8s.Exec`; stdin is a `PassThrough`, stdout/stderr share one `Writable` so output interleaves like a real terminal, and that same stream exposes `rows`/`columns` so the client-node resize channel activates. Sessions are tracked in an `execSessions` Map and aborted on window close, alongside existing log sessions.
  - `preload.js`: exposes `startExec`/`execWrite`/`execResize`/`stopExec` plus disposer-returning `onExecData`/`onExecExit`, following the same subscribe-before-start pattern as pod logs to avoid dropping the first bytes.
  - `renderer/app.js`: `startManageExec`/`stopManageExec` wire an `xterm.js` `Terminal` + `FitAddon` to the session, with a `ResizeObserver` keeping the PTY size in sync; wired into the same teardown choke point as log streams (tab switch, drawer close, leaving the workspace, app quit).
  - `renderer/vendor/xterm/` — vendored `@xterm/xterm` + `@xterm/addon-fit` browser builds (no bundler in this app); verified via `electron-builder --dir` that they ship inside `app.asar`.
- **K8s Manage (Phase 3)** — CPU/memory metrics with hand-drawn SVG sparklines/charts (no charting library):
  - Pods and Nodes tables gain live CPU/Memory sparkline columns; the detail drawer gains a "Metrics" tab with a larger CPU chart and Memory chart for the selected resource.
  - `main.js`: `get-metrics` IPC handler wraps `k8s.Metrics` (`getPodMetrics`/`getNodeMetrics`), parsing CPU (`n`/`u`/`m`/core suffixes) and memory (`Ki`/`Mi`/`Gi`/etc.) into plain millicores/bytes; on any failure — most commonly metrics-server not being installed — it returns `{ok:false, reason:'metrics-server-unavailable', error}` instead of throwing.
  - `preload.js`: exposes `getMetrics`.
  - `renderer/app.js`: polls every 10s while viewing Pods/Nodes, keeping a 60-point ring buffer per resource name in `state.manage.metricsSeries`; `renderSparkline()` draws a filled `<path>` line chart into any container and is shared by both the compact table sparklines and the bigger drawer charts. A failed fetch stops the metrics poll and shows a "Metrics server not installed or unreachable" notice in the drawer, without touching the separate resource-list poller. Wired into the same teardown choke point as the resource poller (kind switch, context/namespace change, leaving the workspace).
