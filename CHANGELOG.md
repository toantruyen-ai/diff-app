# Changelog

## Unreleased

### Refactored & Architected
- **Enterprise-Grade Modular Architecture & Test Suite**: Refactored the codebase to support enterprise-scale growth (designed to scale beyond LENS) with a clean, decoupled, layered architecture and token-efficient AI editing structure:
  - **Modular Main Process (`src/main/`)**: Decomposed the 130KB monolithic `main.js` into single-responsibility services (`k8sService`, `envResolverService`, `azureService`, `resourceActionService`, `watchService`, `podLogService`, `podExecService`, `portForwardService`, `metricsService`, `auditService`, `eventsService`, `kubeconfigStoreService`), utilities (`unitParser`, `resourceFormatter`, `k8sHelper`, `timeout`), constants (`k8sConstants`), and thin IPC handler registration modules (`src/main/ipc/`).
  - **Modular Preload & Renderer (`src/preload/`, `src/renderer/`)**: Isolated preload bridge to `src/preload/index.js` and extracted client utilities (`htmlUtils`, `yamlHighlighter`, `envDiffComputer`, `api/index.js`).
  - **AI Vibe Code Token Optimization**: Reduced average target file size to < 200 lines, enabling AI assistants to inspect and modify single-domain modules without reading 130KB/215KB monoliths, reducing token usage by > 90%.
  - **Comprehensive Vitest Unit Test Suite**: Added Vitest test framework (`vitest.config.js`) and 36 unit tests across 13 test files covering parsers, formatters, services, and IPC contracts with 100% test pass rate.
  - **Renderer modularization made real (esbuild bundling + de-duplication)**: the extracted `src/renderer/utils/*` (`htmlUtils`, `yamlHighlighter`, `envDiffComputer`) were previously dead code — `renderer/app.js` still shipped its own duplicate copies of every one of those functions (`escHtml`, `computeEnvDiffRows`, `maskValue`, `getSourceClass`, `formatSourceLabel`, `highlightYaml*`, `csvEscape`, `rowsToCsv`, `downloadTextFile`), and `index.html` loaded the monolith directly, so the unit tests validated code the app never ran. Wired the shared utils into `app.js` via `require`, deleted the ~200 lines of in-file duplicates, and introduced an **esbuild** bundle step (`npm run build:renderer` → `renderer/dist/app.js`, git-ignored) run automatically by `prestart`/`predev`/`prebuild:*`. `index.html` now loads `dist/app.js`. Call sites of `computeEnvDiffRows` pass filter/search/mask via an options object instead of reading module globals.
  - **DB layer relocated into `src/`**: moved `audit-db.js` → `src/main/db/auditDb.js` and `events-db.js` → `src/main/db/eventsDb.js` (they were the last modules still living at the repo root), updated the service `require` paths, and dropped their now-redundant explicit entries from `build.files` (covered by `src/**/*`; `events-db.js` had in fact been missing from the packaged file list).

- **"Missing metadata.resourceVersion — reload the YAML and try again." on every Edit → Save & Apply**: `get-resource-yaml`/`get-custom-resource-yaml` unconditionally deleted `metadata.resourceVersion` from the returned YAML regardless of the `opts.forEdit` flag, even though `forEdit` exists specifically to keep it for optimistic-concurrency on apply (Phase 14). So the edit textarea never actually contained a `resourceVersion`, and `apply-resource-yaml`/`apply-custom-resource-yaml`'s required-field check always failed. Both handlers now only strip `resourceVersion` when `!opts.forEdit` (the read-only view is unaffected — still resourceVersion-free).
- **Electron Preload Script Loading Failure after Refactoring**: Fixed `Unable to load preload script: ... module not found: ./src/preload/index` and the resulting `Cannot read properties of undefined (reading 'onUpdateAvailable')` error in the renderer. The refactored `preload.js` used a relative `require('./src/preload/index')` which was blocked by Electron's default sandboxing. Disabled sandboxing (`sandbox: false` in `webPreferences` in `src/main/index.js`) to allow preload relative require loading. Also added `src/**/*` to `build.files` in `package.json` to prevent crashes in the packaged application due to missing source files.
- **Missing IPC Handler for 'get-token-expiry'**: Fixed `Error: No handler registered for 'get-token-expiry'` thrown when the renderer queried Azure access token expiration. Re-implemented the `getTokenExpiry` function in `src/main/services/azureService.js` to execute `az account get-access-token` and extract expiration time, registered the handler in `src/main/ipc/azureHandler.js`, and added unit tests in `azureService.test.js`.
- **Audit history (Recycle Bin / History tab) went empty after restarting the app for AKS-picked clusters**: the audit trail's `clusterId` was an MD5 hash of the raw kubeconfig `ref` string. For "Use Local kubeconfig" that's a stable file path, but for a cluster picked via the AKS browser, `ref` is an in-memory-only sequential id (`aks:1`, `aks:2`, …) minted fresh by `storeAksKc()` every time the Electron main process starts — so it never matched between sessions, and every previously-written audit row became permanently unreachable under the new session's `clusterId` (the rows themselves were never lost, just orphaned). Fixed by resolving the actual cluster API server URL (`kc.getCurrentCluster().server`) via a new `resolveClusterId()` helper and hashing that instead — stable across restarts regardless of how the kubeconfig was obtained. Falls back to the raw `ref` if cluster resolution fails, so audit recording never hard-fails because of this.
- **YAML tab stuck on "Loading…" after a successful Save & Apply**: `saveManageYamlEdit()` updated the hidden `<textarea>`'s value on success but never the visible `<pre>` output element, which still held the "Loading…" placeholder text set when edit mode was entered — so switching back to view mode showed "Loading…" forever instead of the freshly applied YAML. Now both are updated on success.

### Added
- **K8s Manage — Recycle Bin**: restore a deleted resource from the audit trail. A new **♻️ Recycle Bin** sidebar entry (alongside Overview) lists every resource whose most recent audit entry is a `delete`, per cluster/namespace, with who deleted it and when. **View YAML** shows the last saved manifest before deletion; **Restore** recreates the resource from it.
  - Restore is a genuine **create**, not the existing edit-restore's replace/PUT — the object no longer exists, so there's no live `resourceVersion` to key a PUT off. Before recreating, server-managed and binding fields are stripped from the stored manifest (`resourceVersion`, `uid`, `creationTimestamp`, `generation`, `managedFields`, `selfLink`, `ownerReferences`, `status`) so the new object isn't rejected or immediately garbage-collected by a stale/gone owner.
  - Scoped to a curated set of top-level kinds plus CRDs — Deployments, StatefulSets, DaemonSets, Services, Ingresses, ConfigMaps, Jobs, CronJobs, PVCs, HPAs, Namespaces, RBAC kinds, NetworkPolicies, StorageClasses, ResourceQuotas, LimitRanges. Deliberately excludes Pods/ReplicaSets (their controller recreates them), Events (ephemeral), Nodes/PVs (cluster/cloud-provisioned) — recreating any of these is meaningless or actively harmful.
  - **Secrets are never restorable**: their `data` values are stored redacted (`***REDACTED***`) in the audit trail at delete time, so there's nothing real to recreate from — the Restore button is disabled with an explanatory tooltip instead of silently producing a broken Secret.
  - Recreating an already-existing name surfaces a clear "already exists" conflict instead of a raw API error. The recreate itself is recorded as a new `restore` audit entry, same as the existing edit-restore flow.
  - `audit-db.js`: `getDeletedResources()` — one row per resource identity (its latest audit entry, if that entry is a delete), so a delete → restore → delete cycle doesn't show duplicate entries. `main.js`: `createManageObject()` (create-side counterpart to the existing `readManageObject()`), `stripForRecreate()`, and `restore-deleted-resource` / `get-deleted-resources` IPC handlers.

Manual entry of SQL server or database names is not permitted to enforce corporate rules. All Edit/Delete/Scale/Restart/Cordon/Bulk actions are **locked by default** and only unlocked when Audit is successfully connected; a 🔒/🔓 badge in the sidebar header shows the current state.
  - **Audit logging**: every YAML edit (`apply-resource-yaml`, `apply-custom-resource-yaml`) and delete (`resource-action`, `custom-resource-action`) now captures the before/after YAML into a `k8senvdiff_audit` table, with cluster, namespace, kind, name, edit version, and the `az account show` user identity. Secrets are stored redacted. All 20+ built-in kinds and all CRDs are covered.
  - **History tab**: new drawer tab showing the audit trail for the selected resource — each version shows action type (edit/delete/restore), who made the change, and when. "Diff" shows a color-coded YAML diff (reusing the existing `Diff.diffLines` renderer); "Restore" replaces the live resource with a previous version's YAML (using optimistic-concurrency PUT) and records its own audit entry.
  - `audit-db.js`: new module mirroring `events-db.js` pattern — singleton mssql pool, `discover()` via `az sql server list`, `connect()`, `ensureSchema()`, `insertAudit()`, `getVersions()`, `getVersionYaml()`, `nextEditVersion()`.
  - Credential overlay: dedicated popup (same styling as the existing confirm modal) for entering SQL username/password; auto-connects on restart from saved credentials.
  - **Clean YAML & Diff Views**: strips auto-generated clutter (metadata fields `resourceVersion`, `uid`, `creationTimestamp`, annotation `kubectl.kubernetes.io/last-applied-configuration`, and root-level `status` block) from all read-only and edit YAML views on the fly. On YAML edits, `resourceVersion` is passed out-of-band to ensure optimistic concurrency conflict protection remains active.
  - **Manual Azure CLI Re-authentication**: Added a `btn-token-reauth` (`🔄`) button in the titlebar next to the Token countdown, executing `az-logout` and launching the login overlay for quick recovery from expired credential sessions.

- **K8s Manage (Phase 16)** — real-time updates via k8s watch streams, replacing the poll timer for high-churn kinds:
  - Pods, Deployments, ReplicaSets, StatefulSets, DaemonSets, Jobs, and Events now stream live — a `kubectl scale`/delete/etc. from outside the app shows up in the table in well under a second, instead of waiting for the next 5s poll tick. Every other kind (Services, ConfigMaps, Secrets, RBAC/Policy kinds, Nodes, PVs/PVCs, HPAs, CronJobs, Namespaces) and all CRDs stay on the existing poll timer — deliberately not attempted this phase, since they're low-churn enough that the added complexity isn't worth it yet. Metrics polling and the Overview digest are both unaffected (metrics-server has no watch API; the digest stays a one-shot fetch).
  - `main.js`: `listKindItems` now returns the full list response body (not just `.items`) so its `resourceVersion` is available to seed a watch; new `watch-start`/`watch-stop` handlers do an initial LIST to snapshot the table, then open a `k8s.Watch` stream from that point forward, forwarding ADDED/MODIFIED/DELETED events over per-session channels — the same Map-keyed-by-session-id pattern as the existing pod-log/exec/port-forward sessions, torn down on explicit stop and on window close. On disconnect (network blip, server-side timeout, or a `410 Gone` from a stale resourceVersion), it reconnects with exponential backoff and **always re-lists from scratch** rather than trying to resume — deliberately simpler and avoids an entire class of "missed delta after reconnect" bugs, at the cost of one extra LIST call on the rare occasions a reconnect actually happens. If the very first connection attempt fails outright (most likely the `watch` RBAC verb is denied even though `list`/`get` are allowed), it's treated as permanent and the renderer falls back to polling instead of retrying forever against something that can't be watched.
  - `preload.js`: exposes `startWatch`/`stopWatch` plus disposer-returning `onWatchSync`/`onWatchEvent`/`onWatchError`.
  - `renderer/app.js`: `startManageWatch`/`stopManageWatch` wire into every place that used to start/stop the poll timer for the selected kind (kind switch, namespace/context change, leaving Manage, app close); watch deltas apply directly onto `state.manage.rows` and re-render through the *existing* diff-by-key table patcher, so no new DOM update logic was needed. Respects the existing Auto-Poll toggle — with it off, a watch-enabled kind gets the same one-shot-only behavior Tier-2 kinds already had, rather than watching unconditionally.

### Performance
- **Diff-update renderer**: `renderManageTable` now diffs rows by key and patches only changed cells in-place, reducing DOM mutations by ~95% during steady-state polling refreshes. Full DOM rebuild only occurs on kind/namespace switches.
- **API Client caching**: `getCachedApiClients()` in `main.js` caches `KubeConfig` parsing and all 7 API client instantiations (CoreV1, AppsV1, BatchV1, etc.) per `(ref, context)` with a 5-minute TTL, eliminating redundant YAML parsing and object creation on every poll cycle.
- **Resource rows cache**: A per-`(namespace, kind)` in-memory cache stores previously fetched rows. Switching between kinds now shows the cached data instantly while the background HTTP request completes, eliminating the blank-table delay. Cache is invalidated on context/namespace change.
- **Debounced kind navigation**: Rapid clicks across multiple sidebar kinds within 100ms only fire a single IPC/HTTP request for the final kind. Cached data renders immediately on each click; only the HTTP call is debounced.
- **Virtual scrolling (>200 rows)**: When filtered row count exceeds 200, `renderManageTable` switches to a virtual scroll renderer that only creates DOM nodes for rows visible in the viewport plus a 10-row buffer above/below. Two spacer `<tr>` elements maintain correct scroll height and scrollbar position. Scroll events are throttled via `requestAnimationFrame`. Below the threshold, the existing diff-update path is used unchanged.
